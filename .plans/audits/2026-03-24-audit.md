## Critical Findings

### C1. `agentObservationTriggerSchema` missing `safe-add-owner-requested` trigger [NEW]
- **File**: `packages/shared/src/contracts/schema.ts:452-468`
- **Issue**: `coop.ts:318` emits an agent observation with `trigger: 'safe-add-owner-requested'`, but this value is not in the `agentObservationTriggerSchema` enum. The enum currently has 15 values; this 16th is missing.
- **Impact**: TS2322 compile error. At runtime, Zod validation will reject the observation, silently dropping the "trusted member needs Safe co-signer" workflow trigger. This affects the onchain member onboarding flow — trusted members joining a coop will not get their Kernel accounts added as Safe owners.
- **Recommendation**: Add `'safe-add-owner-requested'` to `agentObservationTriggerSchema` in `schema.ts`.

## High Findings

### H1. `dashboard.ts` uses `ReviewDraft` type without importing it [NEW]
- **File**: `packages/extension/src/background/dashboard.ts:255,346`
- **Issue**: The `buildSummary()` return type and `writePopupSnapshot()` parameter reference `ReviewDraft`, but the type is never imported from `@coop/shared`. This was introduced in an uncommitted change that refactored draft title resolution.
- **Impact**: TS2304 "Cannot find name 'ReviewDraft'" — the type resolves to `any`, defeating type safety for the entire dashboard summary pipeline. Parameters `a`, `b`, `d` in `toSorted`/`map` callbacks all become implicit `any`.
- **Recommendation**: Add `type ReviewDraft` to the import from `@coop/shared` at line 1.

### H2. `dashboard.ts` uses `.toSorted()` but tsconfig lib is ES2022 [NEW]
- **File**: `packages/extension/src/background/dashboard.ts:349`
- **Issue**: `Array.prototype.toSorted()` is an ES2023 method. The base tsconfig specifies `"lib": ["ES2022", "DOM", "DOM.Iterable"]`. TypeScript reports TS2550.
- **Impact**: The code works at runtime (modern browsers support `toSorted`), but the type checker does not recognize the method. Combined with H1, this degrades the entire dashboard type chain.
- **Recommendation**: Either change lib to `ES2023` in `tsconfig.base.json`, or use `[...drafts].sort(...)` instead.

### H3. `capture.ts` uses `tab.url` after guard but without type narrowing [NEW]
- **File**: `packages/extension/src/background/handlers/capture.ts:130,138,146,165`
- **Issue**: `runCaptureForTabs` iterates `chrome.tabs.Tab[]` and calls `isSupportedUrl(tab.url)` as a guard (line 126), then uses `tab.url` at 4 subsequent call sites. Since `isSupportedUrl` doesn't use a type predicate (`url is string`), TypeScript still sees `tab.url` as `string | undefined`, producing 4 TS2345 errors.
- **Impact**: Type safety gap. A refactor that removes or reorders the guard could introduce a runtime crash on `new URL(undefined)`.
- **Recommendation**: Either make `isSupportedUrl` a type guard (`function isSupportedUrl(url: string | undefined): url is string`), or add an explicit `if (!tab.url) continue;` before the first use.

### H4. `ChickensTab.tsx` accesses non-existent `origin` property on `Artifact` [NEW]
- **File**: `packages/extension/src/views/Sidepanel/tabs/ChickensTab.tsx:326,329,333`
- **Issue**: The component accesses `artifact.origin?.sourceUrls?.[0]` but the `Artifact` type has `originId: string`, not `origin: { sourceUrls: string[] }`. TypeScript suggests `originId` as the closest match (TS2551).
- **Impact**: The source URL link in the Chickens tab shared items section will never render — `artifact.origin` is always `undefined`, so the conditional always evaluates to `null`.
- **Recommendation**: Either add an `origin` relation to the Artifact type, or use `artifact.sources[0]?.url` which is available on the schema.

### H5. `setCoopArchiveSecrets` still receives optional `proofs` [STILL OPEN, 2 cycles]
- **File**: `packages/extension/src/background/handlers/archive.ts:628`
- **Issue**: The `proofs` field in `payload.secrets` is `string[] | undefined`, but `setCoopArchiveSecrets` requires `proofs: string[]` (non-optional). The spread of `coopId` was added since last audit but doesn't fix the `proofs` issue.
- **Recommendation**: Default proofs: `{ ...payload.secrets, coopId: payload.coopId, proofs: payload.secrets.proofs ?? [] }`.

### H6. `onchainState` initialized as `{}` [STILL OPEN, 2 cycles]
- **Files**: `packages/extension/src/views/shared/useCoopActions.ts:96`, `packages/extension/src/views/Sidepanel/hooks/useCoopForm.ts:100,144`
- **Issue**: Both files initialize `onchainState` as `{}`. The required type needs `chainId`, `chainKey`, `safeAddress`, `statusNote`, `safeCapability`. Additionally, `useCoopForm.ts:144` accesses `.statusNote` on the empty object (TS2339).
- **Recommendation**: Initialize with defaults: `{ chainId: 0, chainKey: 'sepolia', safeAddress: '', statusNote: '', safeCapability: 'unavailable' }`.

### H7. `SidepanelApp.tsx` compares plan status to impossible value `'proposed'` [NEW]
- **File**: `packages/extension/src/views/Sidepanel/SidepanelApp.tsx:144`
- **Issue**: `agentDashboard?.plans?.filter((p) => p.status === 'proposed')` — the `AgentPlanStatus` type is `'pending' | 'approved' | 'rejected' | 'executing' | 'completed' | 'failed'`. The value `'proposed'` never matches, so this filter always returns 0 results (TS2367).
- **Impact**: The Nest tab badge count never includes agent plans, even when plans are awaiting approval. The correct status to check is likely `'pending'`.
- **Recommendation**: Change to `p.status === 'pending'`.
