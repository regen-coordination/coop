## Medium Findings

### M1. ZK proof generation hardcoded to `null` [DEFERRED, 5 cycles]
- **File**: `packages/extension/src/background/handlers/review.ts:66`
- **Issue**: Anonymous publish sets `membershipProof = null` with TODO. Snarkjs WASM workers require `URL.createObjectURL` and `new Worker`, both unavailable in MV3 service workers.
- **Status**: DEFERRED. The fix requires either (a) offscreen document delegation for ZK operations, or (b) a different ZK library compatible with MV3 constraints.
- **Rationale**: Structural platform limitation, not a code quality issue. Anonymous publishing works -- it just skips the proof. Stops escalation.
- **Note**: Per escalation rules, 5+ cycles requires Executive Summary mention. This finding is DEFERRED so escalation does not apply, but it is flagged here for visibility as a **chronic deferred item**.

### M2. Landing page unused exports [NEW]
- **File**: `packages/app/src/views/Landing/landing-data.ts:139,150`
- **Issue**: `partnerMarks` and `heroSignalFragments` are exported but never imported anywhere in the codebase. knip confirms them as unused exports.
- **Impact**: Minor dead code. These appear to be remnants from an earlier landing page iteration.
- **Recommendation**: Remove the `export` keyword or delete the constants if no longer needed.

### M3. `app/src/app.tsx` approaching god object threshold [NEW]
- **File**: `packages/app/src/app.tsx` (833 lines)
- **Issue**: The main app component file contains route resolution, dev environment management, all receiver views, error boundary, and multiple inline components. This file grew from the prior audit (was not in the top offenders list).
- **Impact**: Maintainability. Adding new routes or receiver features further inflates this file.
- **Recommendation**: Extract `ErrorBoundary`, `DevGate`, and route resolution into separate files. The receiver views are already separate components -- the orchestration glue in `app.tsx` could be split into a `ReceiverApp` component.
