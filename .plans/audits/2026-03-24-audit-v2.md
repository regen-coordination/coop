## High Findings

### H1. `onchainState` type not narrowed from `sendRuntimeMessage` response [STILL OPEN, 3 cycles]
- **File**: `packages/extension/src/views/shared/useCoopActions.ts:96`, `packages/extension/src/views/Sidepanel/hooks/useCoopForm.ts:100,144`
- **Issue**: Both `resolveOnchainState()` functions call `sendRuntimeMessage({ type: 'resolve-onchain-state' })` and return `response.data`. The response type is generic (`RuntimeActionResponse`), so `data` resolves to `{}` instead of `OnchainState`. TypeScript produces TS2739 at the call sites and TS2339 when accessing `.statusNote` at `useCoopForm.ts:144`.
- **Impact**: 3 type errors. At runtime works because the data IS an `OnchainState`, but the type system can't verify it.
- **Recommendation**: Cast `response.data as OnchainState` in `resolveOnchainState()`, or add discriminated return types to `sendRuntimeMessage` keyed by message type.

### H2. JSX namespace missing in 5 Popup/Sidepanel files [STILL OPEN, 3 cycles, escalated from MEDIUM]
- **Files**: `PopupFooterNav.tsx:50`, `PopupShell.tsx:12,13,15`, `TabStrip.tsx:104`
- **Issue**: 5 `Cannot find namespace 'JSX'` errors (TS2503). The app package resolved this, but extension still has files where JSX types are not resolved, particularly in Popup components.
- **Recommendation**: Add `/// <reference types="react/jsx-runtime" />` to affected files, or add `"jsx": "react-jsx"` + `"types": ["react/jsx-runtime"]` to extension tsconfig.

### H3. `member-account.ts` has 12 type errors for passkey/greenGoods nullability and state shape [NEW]
- **File**: `packages/extension/src/background/handlers/member-account.ts` (lines 88, 128, 209, 276, 290, 302, 346, 363, 388, 425, 439, 464)
- **Issue**: 6 `authSession.passkey` possibly undefined (TS18048), 4 `coop.greenGoods` possibly undefined (TS18048), 2 `saveState(nextState)` shape mismatches (TS2345). The `CoopSharedState` type was refined to make `passkey` and `greenGoods` optional, but the handler code does not guard for undefined.
- **Impact**: 12 type errors. Runtime safety depends on implicit assumptions that passkey and greenGoods are always present when these handlers are called, which is fragile.
- **Recommendation**: Add early-return guards: `if (!authSession.passkey) return { ok: false, error: 'Passkey required' }` and `if (!coop.greenGoods) return { ok: false, error: 'Green Goods not configured' }`.

### H4. `NestInviteSection.tsx` calls `useRef()` with 0 arguments (React 19 types) [NEW]
- **File**: `packages/extension/src/views/Sidepanel/tabs/NestInviteSection.tsx:37`
- **Issue**: `useRef<ReturnType<typeof setTimeout>>()` -- React 19 `@types/react` (v19.2.14) requires an initial value argument for `useRef`. TS2554: "Expected 1 arguments, but got 0."
- **Impact**: Single type error. Works at runtime but will break if strict type checking is enforced in CI.
- **Recommendation**: Change to `useRef<ReturnType<typeof setTimeout>>(undefined)`.

### H5. `useSidepanelOrchestration.ts` passes untyped `captureMode` string [NEW]
- **File**: `packages/extension/src/views/Sidepanel/hooks/useSidepanelOrchestration.ts:324`
- **Issue**: `captureMode` is typed as `string` but the message payload expects `CaptureMode` literal union (`"manual" | "5-min" | "10-min" | "15-min" | "30-min" | "60-min"`). TS2322.
- **Impact**: Type safety gap. If a non-CaptureMode string is passed, it would be accepted at compile time.
- **Recommendation**: Cast or validate: `captureMode: patch.captureMode as CaptureMode`.
