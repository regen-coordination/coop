## High Findings

### H1. `closedConnections` Set grows unboundedly (memory leak) [NEW]
- **File**: `packages/api/src/ws/handler.ts:43,62`
- **Issue**: The `closedConnections` Set tracks raw WebSocket keys for connections that have closed (to reject late messages). Keys are added in `cleanup()` at line 62 but are **never removed**. On a production signaling server handling thousands of connections over days/weeks, this set grows without bound.
- **Impact**: Gradual memory growth proportional to total historical connections. The keys are `ws.raw` references (Bun ServerWebSocket objects), which may also prevent GC of associated buffers.
- **Confidence**: HIGH (verified -- no `.delete()` or `.clear()` on `closedConnections` anywhere in source)
- **Recommendation**: Either (a) use a `WeakSet` instead of `Set` (if `ws.raw` is always an object, which it is for Bun), or (b) add a periodic sweep that removes stale entries, or (c) remove entries after a short TTL (e.g., 60s post-close is sufficient to reject late messages).

### H2. Dockerfile builds without lockfile (non-deterministic production deps) [NEW]
- **File**: `packages/api/Dockerfile:3`
- **Issue**: `COPY bun.lock* ./` uses a glob that matches nothing -- the API package has no local `bun.lock` (it lives at the monorepo root). The subsequent `bun install --production` runs without a lockfile, meaning dependency versions are resolved fresh from registries on every build.
- **Impact**: Production builds may get different dependency versions between deployments. A patch release of `hono`, `yjs`, `lib0`, or `y-protocols` could introduce breaking changes silently.
- **Confidence**: HIGH (verified -- no `bun.lock` in `packages/api/`, root lockfile at `bun.lock`)
- **Recommendation**: Copy the root `bun.lock` into the Docker build context. If using `docker build -f packages/api/Dockerfile .` from repo root, change to `COPY bun.lock ./`. If building from the package directory, restructure to a multi-stage build that copies the root lockfile.

## Medium Findings

### M1. 9 type errors in test files [NEW]
- **Files**: `packages/api/src/ws/__tests__/live-signaling.test.ts:10,41`, `packages/api/src/ws/__tests__/live-sync.test.ts:12`, `packages/api/src/ws/__tests__/yjs-sync.test.ts:85,118,369,404,431,454`
- **Issue**: Two categories:
  1. **Missing `@types/ws`**: The `ws` module (devDep) has no types installed. Lines 10, 41 of `live-signaling.test.ts` and line 12 of `live-sync.test.ts` produce `TS7016` (implicit any) and `TS7006` (parameter implicit any).
  2. **`Uint8Array | undefined` narrowing**: 6 errors in `yjs-sync.test.ts` where `ws.sent[n]` could be `undefined` (due to `noUncheckedIndexedAccess: true` in tsconfig) but is passed directly to functions expecting `Uint8Array`.
- **Impact**: Tests run fine (Vitest doesn't type-check), but `tsc --noEmit` fails for the package. This blocks CI type-check gates.
- **Recommendation**: (a) Add `@types/ws` to devDependencies. (b) Add non-null assertions or guards for `ws.sent[0]!` accesses in yjs-sync tests.

### M2. No authentication or origin validation on WebSocket endpoints [NEW]
- **File**: `packages/api/src/ws/index.ts:16-24`, `packages/api/src/ws/index.ts:32-61`
- **Issue**: Both the signaling (`/`) and Yjs sync (`/yws/:room`) WebSocket endpoints accept connections from any origin with no authentication. Any client can subscribe to any topic or join any room.
- **Impact**: For the current architecture (local-first, end-to-end encrypted sync), this is acceptable -- the server is an untrusted relay. However, it means: (a) anyone can create rooms/topics and consume server resources, (b) no rate limiting on connections themselves (only on blob relay requests within a room), (c) topic names and room names are enumerable by an attacker who can guess them.
- **Confidence**: HIGH (verified -- no CORS middleware, no auth checks, no origin header validation)
- **Recommendation**: Consider adding: (a) connection-level rate limiting (max connections per IP), (b) room name validation (require UUID format or signed tokens), (c) CORS headers for the HTTP health endpoint. Mark as ACCEPTED if the untrusted-relay model is intentional.

### M3. `loadRoomState` race condition on room creation [NEW]
- **File**: `packages/api/src/ws/yjs-sync.ts:182-183`
- **Issue**: `loadRoomState()` is called with `void` (fire-and-forget) in `getOrCreateRoom()`. The room is returned immediately with an empty Y.Doc, and the first connecting client receives sync step 1 based on this empty state. The persisted state loads asynchronously and arrives as a later update.
- **Impact**: If persisted state exists, the first client in a room after a server restart will momentarily see an empty document, then receive the persisted state as an update. For Yjs CRDTs this resolves correctly (merge), but there's a window where a client could make edits against the empty doc that conflict with the about-to-load persisted state. In practice, this is unlikely to cause data loss due to CRDT semantics, but it's architecturally unsound.
- **Confidence**: MEDIUM (CRDT merge mitigates the practical impact, but the race is real)
- **Recommendation**: `await loadRoomState()` before returning the room, or defer sending sync step 1 until loading completes. This would require making `getOrCreateRoom` async.

## Low Findings

### L1. DRY violation: `rawKey`/`wsKey` function duplicated 3 times
- `packages/api/src/ws/handler.ts:13` -- `rawKey(ws: WSContext): unknown`
- `packages/api/src/ws/topics.ts:11` -- `wsKey(ws: WSContext): unknown`
- `packages/api/src/ws/yjs-sync.ts:37` -- `rawKey(ws: WSContext): unknown`
- All three have identical logic: `return ws.raw ?? ws`. Extract to a shared `ws-utils.ts`.

### L2. Unused exported types (8 types across 3 files)
- `packages/api/src/lib/env.ts`: `ServerEnv`
- `packages/api/src/ws/types.ts`: `SubscribeMessage`, `UnsubscribeMessage`, `PublishMessage`, `PingMessage`, `SignalingMessage`
- `packages/api/src/ws/yjs-sync.ts`: `YjsRoom`, `YjsSyncHandlerOptions`
- These are well-typed interfaces that document the API contract. `SignalingMessage` and `YjsRoom` are useful for consumers and tests. Consider keeping them as public API types even if not currently imported. `ServerEnv` is only consumed internally by `loadEnv`.

### L3. `send` functions allow CONNECTING state (readyState 0)
- `packages/api/src/ws/handler.ts:24` and `packages/api/src/ws/yjs-sync.ts:48`
- Both `send` functions allow sending when `readyState === 0` (CONNECTING). Per WebSocket spec, `send()` in CONNECTING state throws `InvalidStateError`. The signaling handler catches the error; the Yjs handler silently drops it. In practice, Bun's WS upgrade completes before handler callbacks fire, so this never triggers. Minor code correctness issue.
