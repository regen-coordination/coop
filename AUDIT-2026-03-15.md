## What Works (Verified Real, Not Stubbed)

- Advisory approval mode by default, configurable per skill
- Timeout enforcement (30s default), skip conditions, error recovery

## Known Gaps & Honest Assessment

### Inference Quality
Qwen2.5-0.5B is a 500M parameter model running in-browser. For domain tasks like opportunity extraction and grant fit scoring, output quality is mediocre. The pipeline architecture is sound and ready for better models, but today's outputs need heavy human triage.

### Identity Recovery
Passkey-only with no wallet recovery or backup mechanism. Lose the device, lose access. Acknowledged but unresolved.

### Mobile
MV3 extension is desktop-only. The companion PWA is capture-only, not the full experience.

### Green Goods / ERC-8004 / FVM
Contract calls work, garden minting works, but full lifecycle (pool creation, cross-chain attestation flows, reputation accumulation) is partial. FVM CoopRegistry is new with basic impl only.

### No External Users
Solo-built with no evidence of external user testing. Validation suites test the builder's happy paths, not real user journeys.

### Error Handling Depth
Alpha-grade error recovery. The "never swallow errors" principle is followed but edge case handling is thin at 22 days.

### Hackathon Context
Submitting to PL_Genesis: Frontiers of Collaboration (deadline: March 16, 2026). Filecoin/Storacha and FVM integrations built with this submission in mind.

## Architecture Strengths

1. **Clean module boundaries**: 17 shared modules with barrel exports, no deep imports
2. **Dual-mode everything**: Mock/live toggles for onchain and archive prevent dev friction
3. **Data model is multi-coop-ready**: All tables have coopId scoping, drafts support multi-coop routing
4. **Offline-first**: Works without internet, syncs when connected
5. **No cloud inference dependency**: Entire agent pipeline runs locally
6. **Comprehensive validation suites**: Named suites for different confidence levels
7. **Typed throughout**: Zod schemas for all domain objects, validated at boundaries

## Architecture Risks

1. **Extension as primary surface**: MV3 service worker lifecycle creates reliability challenges for long-running agent cycles
2. **Single Yjs doc per coop**: Could become a bottleneck as coop state grows
3. **WebRTC for sync**: NAT traversal issues, TURN relay optional but not default
4. **No server-side persistence**: All state is browser-local; if IndexedDB is cleared, everything is lost
5. **Passkey vendor lock-in**: Platform authenticator differences across browsers/devices

## Addendum: Multi-Coop UI & Agent Memory (2026-03-15, post-audit)

### Multi-Coop UI/UX
- `CoopSwitcher` component replaces bare `<select>` dropdown: 0 coops = hidden, 1 coop = static label, 2+ coops = pill dropdown with per-coop badge summaries (pending drafts, pending actions)
- `set-active-coop` background handler now refreshes badge and notifies open sidepanels
- `DashboardResponse` includes `coopBadges: CoopBadgeSummary[]` computed per coop

### Agent Memory Persistence
- `queryMemoriesForSkill()` convenience function in shared (merges skill-pattern + observation-outcome + general, deduplicates by ID, caps at limit)
- Memory reads injected into skill prompts via `buildSkillContext()` → `buildSkillPrompt()` (formatted as `Agent memories:` section between knowledge context and source context)
- Memory writes via `extractMemoriesFromOutput()` + `writeSkillMemories()` after successful skill completion (fire-and-forget, never breaks cycle)
- Supported memory-producing schemas: opportunity-extractor, theme-clusterer, review-digest, capital-formation-brief, publish-readiness-check
- Transactional/scoring schemas (Green Goods, ERC-8004, grant-fit-scorer, ecosystem-entity-extractor) produce no memories
- `pruneExpiredMemories()` runs in agent cycle `finally` block
- `AgentMemorySection` in OperatorConsole: collapsible card with type filter buttons, content, domain, confidence, timestamps
- Observation-outcome and skill-pattern memories expire after 30 days; coop-context memories have no expiry
