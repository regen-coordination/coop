# Coop App

`@coop/app` carries two related jobs:

- the public landing and education surface
- the receiver PWA for capture, pairing, inbox, and board viewing

## Route Ownership

The current app owns these first-class routes:

- `/` as a bootstrap route that decides between landing and receiver entry
- `/landing` for the explicit marketing route
- `/pair` for receiver pairing
- `/receiver` for capture
- `/inbox` for local receiver state and queued captures
- `/board/:coopId` for read-only board views

The runtime implements that directly:

- desktop browsers opening `/` are sent to `/landing`
- mobile or installed PWA sessions opening `/` are sent to `/pair` or `/receiver` based on local
  pairing state

## Why The App Exists

The app gives Coop a second product surface without turning the product into two separate systems.
Its main jobs are:

- make mobile capture practical
- let secondary devices contribute context
- provide a board and pairing surface outside the extension
- keep installation and onboarding understandable for non-extension contexts

## Receiver Shell Direction

The receiver shell is intended to feel app-like:

- safe-area aware
- persistent app chrome
- install nudges in the app flow instead of hidden settings-only prompts
- direct support for audio, photo, file, and link capture

## Pairing And Sync

The pairing model depends on a payload generated by the extension. The app stores a local receiver
identity and sync configuration, then queues captures back into the member-private intake path that
the extension converts into drafts or candidates.

The receiver shell uses the same shared sync primitives as the extension: local Dexie storage,
Yjs-backed room state, configured signaling URLs, and the shared WebSocket sync base URL.

The canonical protocol and storage description now lives in
[Receiver Pairing & Intake](/reference/receiver-pairing-and-intake). Read that doc when you are
changing:

- pairing payload shape or expiry rules
- receiver bridge messaging
- HMAC envelope validation
- member-private intake routing into **Nest**
- receiver-origin requirements for Chrome Web Store builds

## Why This Matters For Builders

Historically the docs under-described the app package because the extension was the dominant story.
That is no longer enough. If you are touching capture, pairing, receiver storage, or mobile install
UX, you are working on a core product path, not a sidecar demo surface.
